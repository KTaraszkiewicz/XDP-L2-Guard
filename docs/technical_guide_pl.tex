\begin{abstract}
XDP-L2-Guard to wysokowydajny, programowalny filtr orkiestracji sieciowej wykorzystujący mechanizm Linux eXpress Data Path (XDP). Działając bezpośrednio w punkcie podpięcia (hook) sterownika karty sieciowej (NIC), zapewnia możliwość przetwarzania pakietów w czasie poniżej mikrosekundy dla ruchu w warstwie 2 i 3. Umożliwia to odrzucanie pakietów z prędkością łącza (wire-speed), przekierowywanie pakietów, hairpinning oraz translację adresów sieciowych (NAT) ze śledzeniem stanu przed alokacją buforów gniazd (\texttt{sk\_buff}) przez jądro Linuksa. Niniejszy podręcznik służy jako wyczerpujące, niskopoziomowe odniesienie techniczne obejmujące struktury pamięci na poziomie języka C, zarządzanie CO-RE w języku Python, algorytmy sum kontrolnych oraz strategie wdrażania.
\end{abstract}

\section{eBPF a tradycyjne moduły jądra Linuksa}

Historycznie rzecz biorąc, modyfikowanie lub rozszerzanie zachowania jądra Linuksa wymagało napisania modułu jądra Linuksa (LKM – Linux Kernel Module) lub rekompilacji kodu źródłowego jądra. Chociaż moduły LKM mają pełne uprawnienia i nieograniczony dostęp do pamięci jądra oraz przepływów wykonania, ze swojej natury niosą ze sobą poważne ryzyko. Błąd w module LKM, taki jak dereferencja wskaźnika zerowego lub nieskończona pętla, może natychmiast doprowadzić do paniki jądra (kernel panic), powodując awarię całego systemu operacyjnego. Ponadto moduły LKM stanowią zagrożenie dla bezpieczeństwa, ponieważ działają z nieograniczonymi uprawnieniami Ring 0 bez wbudowanej izolacji.

eBPF (Extended Berkeley Packet Filter) wprowadza zasadniczo inny paradygmat działania. Umożliwia programistom bezpieczne wykonywanie niestandardowego kodu bajtowego (bytecode) wewnątrz jądra bez zmiany kodu źródłowego jądra lub ładowania standardowych modułów. 

\subsection{Zasada działania i uprawnienia}

W przeciwieństwie do tradycyjnego modułu jądra, który jest konsolidowany bezpośrednio z symbolami jądra i wykonywany natywnie jako kod jądra, programy eBPF są wstrzykiwane do jądra za pomocą wywołania systemowego \texttt{bpf()} z przestrzeni użytkownika. Zanim jednak jakikolwiek program eBPF zostanie dopuszczony do wykonania, musi przejść przez rygorystyczny \textbf{Kernel Verifier}. 

Kernel Verifier przeprowadza statyczną analizę kodu bajtowego eBPF, aby zagwarantować, że program jest bezpieczny do uruchomienia. Zapewnia on, że:
\begin{itemize}
    \item Nie ma nieskończonych pętli (wszystkie programy muszą się zakończyć).
    \item Wszystkie dostępy do pamięci są ściśle kontrolowane pod kątem granic zakresu (bounds-checked).
    \item Nie są używane żadne niezainicjalizowane zmienne.
    \item Rozmiar programu nie przekracza rygorystycznych limitów złożoności.
\end{itemize}

Gdy Kernel Verifier zatwierdzi kod bajtowy, \textbf{kompilator JIT (Just-In-Time) eBPF} tłumaczy kod bajtowy na natywne instrukcje maszynowe (np. x86\_64 lub ARM64) specyficzne dla sprzętu hosta. Ten skompilowany przez JIT kod działa następnie z prędkością bliską natywnej bezpośrednio w przestrzeni jądra.

W ten sposób eBPF uzyskuje uprawnienia jądra nie poprzez omijanie warstw bezpieczeństwa, ale poprzez matematyczne udowodnienie jądru, że jego wykonanie jest bezpieczne. Nie może on uzyskać dostępu do dowolnej pamięci jądra; jest ściśle ograniczony do struktur danych specyficznych dla kontekstu (takich jak struktura \texttt{xdp\_md} dla danych pakietów) oraz predefiniowanych funkcji pomocniczych jądra (\texttt{bpf\_helpers}), które zapewniają bezpieczne API do interakcji z szerszym systemem.

\subsection{Przewaga eBPF nad klasycznymi modułami LKM}

\begin{itemize}
    \item \textbf{Bezpieczeństwo i stabilność:} Rygorystyczny proces weryfikacji gwarantuje, że program eBPF nie doprowadzi do awarii systemu. Jeśli kod narusza granice pamięci lub jego wykonanie trwa zbyt długo, weryfikator po prostu odrzuca go w momencie ładowania.
    \item \textbf{Dynamiczne ładowanie i aktualizacje:} Programy eBPF mogą być ładowane, aktualizowane i odinstalowywane w locie, bez restartowania systemu czy przerywania pracy sieci. Umożliwia to dynamiczne i płynne aktualizacje reguł filtrowania pakietów.
    \item \textbf{Wysoka wydajność:} Dzięki kompilacji JIT programy eBPF działają z prędkościami natywnymi. W kontekście XDP pozwala to na przetwarzanie pakietów bezpośrednio w sterowniku karty sieciowej (NIC), zanim jądro w ogóle przydzieli pamięć (\texttt{sk\_buff}), co zapewnia opóźnienie poniżej mikrosekundy.
    \item \textbf{Bezpieczeństwo i izolacja:} Ponieważ eBPF jest ograniczony do określonych punktów podpięcia (hooks) i funkcji pomocniczych, drastycznie zmniejsza obszar podatności na ataki w porównaniu z modułami LKM, które mają absolutną kontrolę nad systemem.
\end{itemize}

System jest ściśle zaprojektowany wokół paradygmatu separacji warstwy sterowania (control plane) od warstwy danych (data plane). \textbf{Warstwa danych (Data Plane)} jest napisana w ograniczonym języku C, skompilowana za pomocą Clang/LLVM do kodu bajtowego eBPF i wykonywana bezpiecznie w kontekście jądra. \textbf{Warstwa sterowania (Control Plane)} wykorzystuje język Python 3 do orkiestracji cyklu życia programu eBPF, używając narzędzi \texttt{iproute2} oraz \texttt{bpftool} do zarządzania asynchronicznymi aktualizacjami stanu.

\section{Implementacja Data Plane (eBPF/XDP)}

Kod warstwy danych (\texttt{src/data\_plane/filter.c}) podczepia się pod pętlę odpytywania NAPI karty sieciowej (NIC) poprzez makro \texttt{SEC("xdp")}. Funkcja \texttt{xdp\_drop\_logic} pełni rolę głównego koordynatora pakietów wejściowych.

\subsection{Układ pamięci i walidacja wskaźników}

Programy eBPF działają w zweryfikowanym środowisku. Zanim weryfikator zezwoli na dostęp do pamięci pakietu, kod must udowodnić, że dostęp w granicach pamięci jest bezpieczny. Program otrzymuje kontekst \texttt{struct xdp\_md}, dostarczający wskaźniki mapowane przez DMA:

\begin{lstlisting}[language=C]
void *data = (void *)(long)ctx->data;
void *data_end = (void *)(long)ctx->data_end;
\end{lstlisting}

Używamy niestandardowego makra \texttt{BOUNDS\_CHECK} zdefiniowanego w pliku \texttt{headers.h}:

\newpage
\begin{lstlisting}[language=C]
#define BOUNDS_CHECK(pointer, type, data_end) \
    do { \
        if ((void *)(pointer) + sizeof(type) > (void *)(data_end)) \
            return XDP_PASS; \
    } while(0)
\end{lstlisting}

Analiza przebiega ściśle warstwa po warstwie:
\begin{enumerate}
    \item \textbf{Warstwa 2 (Ethernet)}: Rzutowanie \texttt{data} na strukturę \texttt{struct ethhdr}. Weryfikacja granic pamięci. Sprawdzenie, czy \texttt{eth->h\_proto == bpf\_htons(ETH\_P\_IP)}. Jeśli nie, przekazanie pakietu do jądra poprzez \texttt{XDP\_PASS}.
    \item \textbf{Warstwa 3 (IPv4)}: Rzutowanie \texttt{data + sizeof(struct ethhdr)} na strukturę \texttt{struct iphdr}. Weryfikacja granic pamięci. Wyodrębnienie docelowego adresu IP (\texttt{ip->daddr}).
\end{enumerate}

\subsection{Action Map Schema}

Wyodrębniony adres IP jest używany jako klucz wyszukiwania w głównej mapie haszującej BPF, \texttt{action\_map}. Mapa oczekuje adresu IPv4 w sieciowej kolejności bajtów (\texttt{\_\_u32}) i zwraca strukturę \texttt{struct action\_cfg}:

\begin{lstlisting}[language=C]
struct action_cfg {
    __u32 target;           // 0: PASS, 1: DROP, 2: TX, 3: REDIRECT, 4: NAT
    __u32 new_ip;           // Docelowy adres IP dla scenariuszy NAT
    __u32 ifindex;          // Indeks interfejsu systemu operacyjnego dla REDIRECT
    __u64 dropped_packets;  // Atomowy licznik wydajnosci
};
\end{lstlisting}

\subsection{Dostępne akcje XDP}

W zależności od pobranej wartości \texttt{target}, jądro wykonuje jedną z następujących procedur:

\subsubsection{XDP\_DROP}
Najprostsza i najszybsza akcja. Sterownik karty sieciowej (NIC) odrzuca ramkę i natychmiast odzyskuje bufor DMA. Kod atomowo inkrementuje globalną mapę tablicy \texttt{total\_dropped} oraz specyficzną dla reguły mapę \texttt{dropped\_packets}.

\subsubsection{XDP\_TX}
\texttt{XDP\_TX} odsyła pakiet z powrotem przez ten sam interfejs, przez który dotarł. W naszej implementacji jest to wykorzystywane do zaawansowanej manipulacji ICMP. W przypadku wykrycia pakietu \texttt{ICMP\_ECHO} (Ping):
\begin{enumerate}
    \item Typ jest nadpisywany jako \texttt{ICMP\_ECHOREPLY} (0).
    \item \textbf{Obliczenia sumy kontrolnej}: Ponieważ Typ 8 zmienił się na Typ 0, 16-bitowa suma kontrolna dopełnienia do jedynki musi być przyrostowo zaktualizowana przez dodanie wartości \texttt{0x0800}.
    \item Adresy MAC i IP są zamieniane miejscami przy użyciu \texttt{\_\_builtin\_memcpy} w celu uzyskania optymalnej kompilacji wbudowanej (inline).
\end{enumerate}

\subsubsection{XDP\_REDIRECT \& ACTION\_NAT (DNAT)}
\texttt{ACTION\_NAT} przeprowadza translację docelowego adresu sieciowego (Destination Network Address Translation – DNAT) bezpośrednio na poziomie sprzętowym. 
\begin{lstlisting}[language=C]
__u32 old_daddr = ip->daddr;
__u32 new_daddr = cfg->new_ip;
__u32 csum = ~bpf_ntohs(ip->check) & 0xFFFF;
csum += (~(old_daddr & 0xFFFF) & 0xFFFF) + (new_daddr & 0xFFFF);
csum += (~(old_daddr >> 16) & 0xFFFF) + (new_daddr >> 16);
while (csum >> 16) csum = (csum & 0xFFFF) + (csum >> 16);
ip->check = bpf_htons(~csum & 0xFFFF);
\end{lstlisting}
Ta logika bitowa przyrostowo aktualizuje sumę kontrolną nagłówka IP bez potrzeby pełnego przeliczania pakietu, zapewniając opóźnienia poniżej mikrosekundy. \texttt{XDP\_REDIRECT} działa według dokładnie tej samej logiki, kończąc się jednak przekazaniem ramki do interfejsu wskazanego w \texttt{dev\_map}.

\section{Warstwa sterowania}

Warstwa sterowania składa się z dwóch głównych skryptów Pythona: \texttt{loader.py} do zarządzania cyklem życia oraz \texttt{guard.py} do definiowania reguł za pomocą interfejsu wiersza poleceń (CLI).

\subsection{Silnik ładujący (\texttt{loader.py})}

Loader obsługuje kompilację typu Ahead-Of-Time (AOT) oraz podłączanie do jądra za pomocą zestawu narzędzi \texttt{iproute2}.

\begin{itemize}
    \item \textbf{Kompilacja}: Uruchamia \texttt{make} w celu skompilowania pliku \texttt{filter.c} do obiektu ELF \texttt{filter.o}.
    \item \textbf{Wybór trybu}: Wykrywa, czy podmontować program w wysokowydajnym trybie natywnym (\texttt{xdpdrv}), czy w ogólnym trybie programowym (\texttt{xdpgeneric}).
    \item \textbf{Podłączanie}: Używa polecenia \texttt{ip -force link set dev <interface> xdpdrv obj filter.o sec xdp}. Flaga \texttt{-force} gwarantuje atomowe nadpisanie nieaktywnych ("zombie") programów.
    \item \textbf{Asynchroniczne odpytywanie}: Działa w nieskończonej pętli (\texttt{time.sleep(1)}), która wywołuje \texttt{bpftool -j map dump name action\_map}. Dekoduje dane wyjściowe JSON, rozpakowuje strumienie bajtów i tłumaczy surowe metryki na czytelne dla człowieka logi.
\end{itemize}

\subsection{Zarządzanie i Dodawanie Reguł do Hash Mapy (\texttt{guard.py} \& \texttt{guard.sh})}

Narzędzia \texttt{guard.py} oraz \texttt{guard.sh} pełnią funkcję interfejsu zarządzającego zbliżonego do \texttt{iptables}. Operują bezpośrednio na mapach eBPF (hashmapach) przechowywanych w pamięci RAM jądra, co pozwala na natychmiastową (bez konieczności restartu) aplikację filtrów. Zapewnia to separację logiki decyzyjnej od samego filtrowania pakietów.

\subsubsection{Mechanizm w Pythonie (\texttt{guard.py})}
Skrypt pobiera z jądra ID map (np. \texttt{action\_map}) za pomocą komendy \texttt{bpftool map list}. Następnie przekłada wyższe abstrakcje (np. adres "192.168.1.100") na postać bajtową zgodną ze strukturą \texttt{action\_cfg} kompilowaną po stronie języka C (z użyciem biblioteki \texttt{struct} i dopasowaniem kolejności bajtów). 

Oto kod pokazujący, jak parametry (cel, zmienione IP, indeks interfejsu itp.) są łączone i dodawane do mapy za pomocą narzędzia \texttt{bpftool}:
\begin{lstlisting}[language=Python]
# Spakowanie struktur (cel, nowe IP, indeks_interfejsu, padding, licznik_dropow) do wartosci Hex
val_hex = " ".join([f"{b:02x}" for b in struct.pack("<IIIIQ", target, new_ip_int, ifindex, 0, 0)])

# Odwolanie przez 'bpftool map update' aby dodac nowa regule (klucz to Docelowe IP w HEX)
res = run_cmd(f"bpftool map update id {act_map_id} key hex {ip_to_hex(target_ip)} value hex {val_hex}")
\end{lstlisting}

Ten mechanizm bezpośredniego odwołania się do konkretnej struktury w mapie BPF gwarantuje czas odczytu (lookup) w złożoności $O(1)$ ze strony eBPF w momencie weryfikacji docelowego IP.

\subsubsection{Mechanizm w Bashu (\texttt{guard.sh})}
Dla środowisk, w których Python jest niedostępny, zbudowaliśmy alternatywę w języku Bash. Aby uzyskać ten sam strumień danych szesnastkowych, kod Basha samodzielnie konwertuje wartości IP oraz liczby używając formatowania binarnego:

\begin{lstlisting}[language=bash]
# 1. Konwersja IP uzytkownika do Big Endian (klucz)
DST_HEX=$(ip_to_hex "$DST_IP")

# 2. Pakowanie wartosci akcji (T_VAL) do 32-bitowego Little Endian
T_HEX=$(pack_u32 "$T_VAL")

# 3. Zlozenie calego bloku pamieci odwzorowujacego strukture C
PAD_HEX="00 00 00 00"
CNT_HEX="00 00 00 00 00 00 00 00"
VAL_HEX="$T_HEX $NAT_HEX $OIF_HEX $PAD_HEX $CNT_HEX"

# 4. Aplikacja konfiguracji za pomoca bpftool na zmapowane identyfikatory (ID map)
for id in $ACT_MAP_IDS; do
    bpftool map update id "$id" key hex $DST_HEX value hex $VAL_HEX
done
\end{lstlisting}

\subsubsection{Przykłady składni CLI}
Oba skrypty współdzielą dokładnie tę samą składnię i argumenty:

\begin{lstlisting}[language=bash]
# Dodaj standardowa regule DROP dla zlosliwego hosta (wersja Python)
sudo python3 guard.py --append --destination 192.168.1.50 --jump DROP

# Ustanow translacje NAT na poziomie sprzetowym (wersja Bash)
sudo ./guard.sh -A -d 10.0.0.100 -j NAT --to-destination 10.0.0.250

# Przekieruj ruch przychodzacy dla adresu 10.0.0.5 na inny wirtualny Ethernet
sudo ./guard.sh -A -d 10.0.0.5 -j REDIRECT --oif veth1
\end{lstlisting}

\section{Działanie w praktyce i testy}

Aby zademonstrować i zweryfikować możliwości systemu XDP-L2-Guard, katalog \texttt{scripts/example/} udostępnia interaktywne środowisko testowe wykorzystujące sieciowe przestrzenie nazw Linuksa (netns).

\subsection{Topologia testowa}
Skrypt \texttt{setup.sh} inicjalizuje symulowane 3-węzłowe środowisko sieciowe:
\begin{itemize}
    \item \textbf{\texttt{ns1} (Filtr/Router)}: Przypisany adres IP \texttt{10.0.0.1}. Obiekt eBPF jest podłączony do wirtualnych interfejsów \texttt{v1-2} i \texttt{v1-3}. Włączone jest przekazywanie pakietów IPv4.
    \item \textbf{\texttt{ns2} (Nadawca)}: Przypisany adres IP \texttt{10.0.0.2}. Działa jako źródło ruchu.
    \item \textbf{\texttt{ns3} (Cel)}: Przypisany adres IP \texttt{10.0.0.3} w trybie nasłuchiwania w celu obserwowania przekierowanych pakietów.
\end{itemize}

\subsection{Dodawanie reguł}
Skrypt \texttt{set\_action.sh} używa skryptu \texttt{guard.sh} do dynamicznego nakładania reguł na ruch pochodzący z węzła \texttt{ns2}:

\begin{lstlisting}[language=bash]
# Terminal 1: Uruchom konfiguracje i utrzymaj aktywny filtr XDP
sudo ./setup.sh

# Terminal 2: Testuj rozne akcje XDP w czasie rzeczywistym
./set_action.sh drop      # Ruch z ns2 jest po cichu odrzucany
./set_action.sh redirect  # Ruch z ns2 jest przekierowywany do ns3
./set_action.sh tx        # Hairpin ICMP echo (Ping odpowiada natychmiast)
./set_action.sh nat       # Docelowy IP jest przepisywany na 10.0.0.3
\end{lstlisting}

\section{Utrzymanie i diagnostyka}

\subsection{Optymalizacja wydajności i ograniczenia sprzętowe}
Aby osiągnąć teoretyczną maksymalną przepustowość (10-40 Gbps), konieczne jest przeprowadzenie kilku optymalizacji na poziomie sprzętowym.

\begin{itemize}
    \item \textbf{Tryb natywny (\texttt{xdpdrv}) a tryb ogólny (\texttt{xdpgeneric})}: Tryb natywny uruchamia program eBPF bezpośrednio w sterowniku karty sieciowej (NIC), zanim jądro przydzieli jakąkolwiek pamięć. Tryb ogólny działa na dalszym etapie stosu. Tryb natywny jest kluczowy dla ochrony przed zalewem pakietów.
    \item \textbf{Wyłączenie sprzętowych odciążeń}: Odciążenia takie jak Generic Receive Offload (GRO) agregują pakiety zanim zostaną one zauważone przez XDP, co powoduje ominięcie filtra. Wyłączenie następuje poprzez: \texttt{sudo ethtool -K eth0 gro off gso off rxvlan off}.
    \item \textbf{Przypinanie rdzeni CPU}: Należy upewnić się, że przerwania IRQ dla XDP są przypięte do dedykowanych fizycznych rdzeni procesora za pomocą \texttt{smp\_affinity}.
\end{itemize}

\subsubsection{Testy trybu natywnego na kartach Intel}
Podczas rzeczywistych testów obciążeniowych wykorzystujących wiele niezależnych maszyn atakujących, które zalewały pakietami interfejs sieciowy Intel, naszym celem było wykorzystanie natywnego trybu XDP (\texttt{xdpdrv}) dla uzyskania maksymalnego przyspieszenia sprzętowego. Ostatecznie napotkaliśmy wyzwania związane z niskopoziomową konfiguracją sprzętową, a nie limity obliczeniowe procesora hosta.

Choć w tym konkretnym teście nie byliśmy w stanie idealnie dopasować konfiguracji sterownika karty sieciowej w celu osiągnięcia przetwarzania z prędkością łącza (wire-speed), uznajemy to za kluczowy obszar przyszłego rozwoju. Jesteśmy w pełni świadomi ogromnego potencjału wydajnościowego, jaki oferuje natywne wykonywanie na poziomie sprzętowym. Byliśmy wyjątkowo blisko działającej natywnej implementacji, a rozwiązanie tych specyficznych dla sprzętu niuansów konfiguracyjnych pozostaje priorytetowym kamieniem milowym dla kolejnej ewolucji XDP-L2-Guard.

\section{Potencjał dla Carrier Ethernet}

Choć standardowa dystrybucja Linuksa nie jest zazwyczaj stosowana jako rdzeniowy router Carrier Ethernet, nowoczesne możliwości jądra dostarczają elementów konstrukcyjnych do testowania tych koncepcji. Carrier Ethernet opiera się na dostarczaniu pakietów o wysokiej przepustowości oraz wewnętrznych mechanizmach etykietowania (takich jak MPLS czy VLAN) w celu efektywnego kierowania ruchem. Nasz projekt, XDP-L2-Guard, służy jako małe środowisko testowe do zrozumienia potencjału eBPF i XDP w tego typu scenariuszach.

\subsection{Routing wewnątrz jądra}

Historycznie routing oparty na etykietach MPLS w systemie Linux opiera się na standardowym stosie sieciowym jądra (poprzez moduły takie jak \texttt{mpls\_router}), co wprowadza narzut na procesor. Nasza implementacja w pliku \texttt{filter.c} skupia się obecnie na podstawowym wyszukiwaniu adresów docelowych IPv4, ale skutecznie demonstruje podstawową zasadę wymaganą dla routera LSR (Label Switch Router).

Działając w punkcie podczepienia \texttt{XDP}, przechwytujemy pakiety przed alokacją \texttt{sk\_buff} przez jądro. Przetestowana przez nas funkcjonalność \texttt{XDP\_REDIRECT} pokazuje, że stworzenie płaszczyzny danych szybkiej ścieżki (fast-path data plane) jest całkowicie możliwe. Choć nie zaimplementowaliśmy pełnego parsowania MPLS, nasz kod udowadnia potencjał: logika eBPF mogłaby łatwo zostać rozszerzona o odczytywanie etykiet MPLS bezpośrednio ze wskaźników \texttt{data} i natychmiastowe kierowanie pakietów za pomocą mapy \texttt{dev\_map}.

\subsection{Akceleracja sprzętowa na kartach Intel}

Przetestowaliśmy XDP-L2-Guard w celu zrozumienia natywnego trybu XDP (\texttt{xdpdrv}) na nowoczesnych kartach sieciowych Intel (takich jak serie 700 oraz 800 ice). Teoretycznie, wykonywanie kodu bajtowego eBPF bezpośrednio w sterowniku kolejki odbiorczej (RX queue) karty sieciowej oferuje wydajność porównywalną z DPDK (Data Plane Development Kit – branżowy standard osiągający wysokie prędkości poprzez całkowite ominięcie jądra systemu operacyjnego na rzecz przetwarzania pakietów w przestrzeni użytkownika), bez konieczności opuszczania środowiska jądra Linuksa.

Mimo że nasz projekt jest jedynie dowodem koncepcji (proof-of-concept), podkreśla on, jak akceleracja sprzętowa (jak Intel Flow Director) połączona z XDP może inteligentnie dystrybuować i przetwarzać ruch przy ekstremalnie niskich opóźnieniach. Nasze eksperymenty potwierdzają teoretyczny potencjał użycia systemu Linux i sprzętu Intel do przetwarzania ruchu klasy operatorskiej (Carrier-grade).

\section{XDP-L2-Guard jako Soft Switch}

Jednym z głównych celów projektu XDP-L2-Guard było zbadanie, w jaki sposób eBPF może służyć jako podstawa dla programowalnego software switcha i jak wypada w porównaniu z tradycyjnymi rozwiązaniami typu firewall.

\subsection{eBPF jako alternatywa dla iptables / nftables}

Tradycyjnie filtrowanie pakietów i NAT w systemie Linux opierają się na narzędziu \texttt{iptables} (które obecnie domyślnie wykorzystuje pod spodem nowszy silnik \texttt{nftables}). Pomimo ulepszeń i optymalizacji w \texttt{nftables}, narzędzia te operują na wyższych warstwach stosu sieciowego, po kosztownej alokacji pamięci dla \texttt{sk\_buff}, co wprowadza zauważalny narzut wydajnościowy. Nasz projekt pozwolił nam praktycznie przetestować korzyści wynikające z działania sprzętowego filtra eBPF.

Dzięki zaimplementowaniu mapy \texttt{action\_map} (\texttt{BPF\_MAP\_TYPE\_HASH}), nasz skrypt XDP osiąga stały czas wyszukiwania $O(1)$. Chociaż nasza warstwa sterowania w Pythonie (\texttt{guard.py}) zarządza jedynie ułamkiem reguł, jakie mogłoby stosować rzeczywiste przedsiębiorstwo, zademonstrowany przez nas podstawowy mechanizm skaluje się bez wysiłku, udowadniając wyższość przetwarzania eBPF XDP nad standardowym stosem \texttt{iptables}/\texttt{nftables}.

\subsection{Prosty NAT i możliwości Data Plane}

Zamiast polegać na stosie netfilter jądra, zaimplementowaliśmy podstawową akcję \texttt{ACTION\_NAT} w języku C, aby przetestować bezpośrednią manipulację pakietami. Nasz kod pomyślnie odczytuje docelowe adresy IP, nadpisuje je i ponownie oblicza sumę kontrolną dopełnienia do jedynki przy użyciu operacji bitowych bezpośrednio w pliku \texttt{filter.c}.

W połączeniu z \texttt{XDP\_REDIRECT} tworzy to podstawowy, ale bardzo szybki przełącznik programowy. Choć XDP-L2-Guard nie jest kompletnym, gotowym do wdrożenia produkcyjnego routerem, z powodzeniem demonstruje niewielką część możliwości eBPF, potwierdzając, że architektura Linuksa ma potencjał do obsługi złożonego, szybkiego routingu i etykietowania na najniższym możliwym poziomie oprogramowania.

\section{Podsumowanie i wnioski}

\subsection{Czego się nauczyliśmy}

Realizacja projektu XDP-L2-Guard dostarczyła bezcennej, praktycznej wiedzy dotyczącej najniższych poziomów stosu sieciowego systemu Linux. eBPF i XDP oferują bezprecedensową wydajność przechwytywania i modyfikowania pakietów, jednak osiągnięcie pełnej prędkości łącza (wire-speed) stawia rygorystyczne wymagania sprzętowe, wymuszając pełne wsparcie dla trybu natywnego ze strony nowoczesnych kart sieciowych (NIC). Pomyślnie udowodniliśmy, że jesteśmy w stanie napisać funkcjonalną warstwę danych zdolną do parsowania nagłówków IP, przeliczania sum kontrolnych oraz orkiestracji podstawowych operacji NAT lub przekierowań – a wszystko to zanim jądro przydzieli jakiekolwiek standardowe bufory gniazd.

\subsection{Znaczenie projektu}

Projekt pozwolił na praktyczne zbadanie mechanizmów stojących za nowoczesnym cloud-native networking oraz Software-Defined Networking (SDN). Implementacja XDP-L2-Guard pokazała możliwości wykorzystania jądra Linuksa jako programowalnego przełącznika o wysokiej wydajności. Rozwiązanie uwydatnia przewagę eBPF nad tradycyjnymi narzędziami, takimi jak \texttt{iptables}. Zdobyta wiedza w zakresie natywnego XDP, operacji na pamięci i separacji Control Plane od Data Plane ma bezpośrednie zastosowanie w nowoczesnej inżynierii sieciowej.

\subsection{Plany na przyszłość}

Choć XDP-L2-Guard służy jako udany dowód koncepcji, istnieje wiele dróg potencjalnej przyszłej eksploracji:
\begin{itemize}
    \item \textbf{Głębokie parsowanie warstwy 2/etykiet:} Rozszerzenie kodu \texttt{filter.c} w celu natywnego parsowania, dodawania i zdejmowania etykiet MPLS lub tagów VLAN, aby zasymulować kompletny router Carrier Ethernet LSR.
    \item \textbf{Zaawansowane strojenie sprzętowe:} Dalsze rozwiązywanie niskopoziomowych niuansów konfiguracji sprzętowej kart sieciowych Intel, aby w pełni odblokować tryb natywny (\texttt{xdpdrv}) i porównać go z rozwiązaniami opartymi na DPDK przy przepustowościach rzędu 40+ Gbps.
    \item \textbf{Śledzenie stanu:} Przejście od bezstanowego DNAT do w pełni stanowych połączeń poprzez zaimplementowanie mechanizmu śledzenia połączeń (conntrack) z wykorzystaniem map haszujących eBPF LRU (Least Recently Used).
    \item \textbf{Wsparcie dla IPv6:} Rozszerzenie bazy kodu C oraz logiki sprawdzania granic pamięci w celu kompleksowej obsługi nagłówków IPv6 oraz protokołów routingu ICMPv6.
\end{itemize}
