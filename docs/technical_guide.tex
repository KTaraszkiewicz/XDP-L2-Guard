\begin{abstract}
XDP-L2-Guard is a high-performance, programmable network orchestration filter leveraging the Linux eXpress Data Path (XDP).
Operating directly inside the Network Interface Card (NIC) driver hook, it provides sub-microsecond packet processing capabilities for Layer-2 and Layer-3 traffic.
This enables wire-speed dropping, packet redirection, hairpinning, and state-aware Network Address Translation (NAT) before the Linux kernel allocates socket buffers (\texttt{sk\_buff}).
This manual serves as an exhaustive, low-level technical reference covering theoretical foundations, system architecture, operational DevOps, and an isolated comprehensive technical chapter detailing concrete codebase implementations, memory structures, Python CO-RE orchestration, and empirical screenshot showcases.
\end{abstract}

\section{eBPF vs. Traditional Linux Kernel Modules}

Historically, modifying or extending the behavior of the Linux kernel required writing a Linux Kernel Module (LKM) or recompiling the kernel source code.
While LKMs have full privileges and unrestricted access to kernel memory and execution flows, they inherently introduce significant risks.
A bug in an LKM, such as a null pointer dereference or an infinite loop, can instantly lead to a kernel panic, crashing the entire operating system.
Furthermore, LKMs pose security risks as they run with unrestricted Ring 0 privileges without built-in isolation.
eBPF (Extended Berkeley Packet Filter) introduces a fundamentally different operational paradigm.
It allows developers to safely execute custom bytecode within the kernel without changing kernel source code or loading standard modules.

\subsection{Method of Operation and Privilege Acquisition}

Unlike a traditional kernel module that is linked directly against kernel symbols and executed natively as kernel code, eBPF programs are injected into the kernel via the \texttt{bpf()} system call from user-space.
However, before any eBPF program is allowed to execute, it must pass through a strict \textbf{Kernel Verifier}.
The Verifier performs static analysis on the eBPF bytecode to guarantee that the program is safe to run.
It ensures that:
\begin{itemize}
    \item There are no infinite loops (all programs must terminate).
    \item All memory accesses are strictly bounds-checked.
    \item No uninitialized variables are used.
    \item The program size does not exceed strict complexity limits.
\end{itemize}

Once the Verifier approves the bytecode, the eBPF \textbf{JIT (Just-In-Time) Compiler} translates the bytecode into native machine instructions (e.g., x86\_64 or ARM64) specific to the host hardware.
This JIT-compiled code then runs at near-native speed directly in kernel space.
Thus, eBPF obtains kernel privileges not by bypassing security layers, but by mathematically proving to the kernel that its execution is safe.
It cannot access arbitrary kernel memory; it is strictly limited to context-specific data structures (like the \texttt{xdp\_md} struct for packet data) and predefined kernel helper functions (\texttt{bpf\_helpers}) that provide a safe API to interact with the broader system.

\subsection{Key Benefits of eBPF over LKMs}

\begin{itemize}
    \item \textbf{Safety and Stability:} The rigorous verification process ensures that an eBPF program cannot crash the system.
If the code violates memory boundaries or takes too long to execute, the Verifier simply rejects it at load time.
    \item \textbf{Dynamic Loading and Updates:} eBPF programs can be loaded, updated, and unloaded on the fly without system reboots or network interruptions.
This enables dynamic and seamless updates to packet filtering rules.
    \item \textbf{High Performance:} Thanks to JIT compilation, eBPF programs run at native speeds.
In the context of XDP, this allows packet processing to occur directly in the NIC driver before the kernel even allocates memory (\texttt{sk\_buff}), yielding sub-microsecond latency.
    \item \textbf{Security and Isolation:} Because eBPF is constrained to specific hooks and helpers, it dramatically reduces the attack surface compared to an LKM that has absolute control over the system.
\end{itemize}

\section{System Architecture and Paradigm}

The system is strictly architected around the paradigm of control/data plane separation.
The \textbf{Data Plane} is designed to execute securely within the kernel context at the lowest possible driver level.
The \textbf{Control Plane} orchestrates the lifecycle of the eBPF program from user-space, utilizing asynchronous state updates to bridge human management commands with kernel-level filtering mechanics.

\subsection{Decoupled Plane Dynamic}

The architectural separation guarantees that the data plane remains highly lightweight and performant.
By removing complex decision-making loops, string parsing, and state management from the kernel space, the XDP program can focus exclusively on high-speed packet inspection and map lookups.
User-space components handle the heavy lifting of configuration parsing, object compilation, map population, and telemetry visualization.

\section{Operational DevOps \& Diagnostics}

Deploying eBPF/XDP applications in real-world environments requires a thorough understanding of kernel attachment modes, driver offloads, and hardware interactions.
This chapter outlines the high-level DevOps considerations and diagnostic frameworks essential for managing XDP-L2-Guard.

\subsection{Performance Tuning \& Real-World Limits}
To achieve theoretical maximum throughput (10-40 Gbps), several hardware-level optimizations are mandatory.
\begin{itemize}
    \item \textbf{Native Mode (\texttt{xdpdrv}) vs. Generic Mode (\texttt{xdpgeneric})}: Native mode runs the eBPF program directly inside the NIC driver, before the kernel allocates any memory.
Generic mode runs later in the stack, acting as a software fallback. Native mode is essential for volumetric flood protection.
    \item \textbf{Disable Hardware Offloads}: Offloads like Generic Receive Offload (GRO) and Large Receive Offload (LRO) aggregate packets in hardware before XDP inspects them, potentially bypassing the filter or exceeding eBPF buffer limits.
These must be suppressed in high-performance deployments.
    \item \textbf{CPU Pinning}: To prevent cache thrashing and lock contention, XDP interrupt request queues (IRQs) should be pinned to dedicated physical CPU cores using \texttt{smp\_affinity}.
\end{itemize}

\subsection{Case Study: Intel NIC Native Mode Testing}
During real-world load testing utilizing multiple independent attacking machines flooding an Intel network interface, our goal was to leverage Native XDP mode (\texttt{xdpdrv}) for maximum hardware acceleration.
Ultimately, we encountered low-level hardware configuration challenges rather than hitting the computational limits of the host CPU.
While we were unable to perfectly align the NIC driver configuration to achieve wire-speed processing in this specific test, we recognize this as a critical area for future development.
We are fully aware of the massive performance potential that native hardware execution offers.
We were exceptionally close to a working native implementation, and resolving these hardware-specific configuration nuances remains a high-priority milestone for the next evolution of XDP-L2-Guard.

\section{Exploring Carrier Ethernet Potential via XDP}

While a standard Linux distribution is not typically used as a core Carrier Ethernet router, modern kernel capabilities provide the building blocks to test these concepts.
Carrier Ethernet relies on high-throughput packet delivery and internal labeling mechanisms (like MPLS or VLANs) to route traffic effectively.
Our project, XDP-L2-Guard, serves as a small-scale testing ground to understand the potential of eBPF and XDP in these scenarios.

\subsection{Testing Internal Routing Capabilities}

Historically, routing based on MPLS labels in Linux relies on the standard kernel network stack (via dedicated kernel modules), which introduces CPU overhead.
While our core architecture focuses on high-speed IP lookups, it effectively demonstrates the underlying principle required for a Label Switch Router (LSR).
By operating at the XDP driver hook, packets are intercepted before reaching the kernel's memory allocation mechanisms.
The fast-path redirection capabilities explored in this project show that it is entirely possible to construct an autonomous data plane.
The architectural concepts proven here could easily be extended to parse MPLS labels directly from raw packet buffers and route frames instantly using dedicated hardware maps.

\subsection{Hardware Offloading and Intel NICs}

We evaluated XDP-L2-Guard with the goal of understanding Native XDP Mode on modern Intel Network Interface Cards (such as the 700 and 800 ice series).
In theory, running eBPF bytecode directly inside the NIC's RX queue driver offers performance comparable to DPDK (Data Plane Development Kit—an industry-standard framework that achieves high speeds by completely bypassing the OS kernel to process packets in user space), without leaving the Linux kernel environment or dedicating entire CPU cores to polling.
Although our project serves as a proof-of-concept, it highlights how hardware acceleration combined with XDP can intelligently distribute and process traffic at extremely low latency.
Our architectural evaluation confirms the theoretical potential of using Linux and modern NIC hardware to process Carrier-grade traffic.

\section{Testing the Software Switch Concept}

One of the primary goals of XDP-L2-Guard was to explore how eBPF can serve as a foundation for a programmable software switch, and how it compares to traditional firewalling tools.

\subsection{eBPF as an Alternative to iptables}

Traditionally, packet filtering and NAT on Linux rely on \texttt{iptables}.
However, \texttt{iptables} uses a sequential list model ($O(n)$ complexity), meaning performance degrades as the rule count grows.
Our project allowed us to evaluate the structural performance benefits of eBPF maps over linear rule evaluation.
By utilizing kernel hash maps, the system achieves constant-time $O(1)$ lookups regardless of the number of active rules.
Although our management control plane manages a subset of the rules a global enterprise might deploy, the fundamental mechanism demonstrated scales effortlessly, proving the viability of eBPF over legacy Netfilter architectures.

\subsection{Basic NAT and the Data Plane Potential}

Instead of relying on the kernel's Netfilter connection tracking stack, we designed a stateless NAT mechanism to evaluate direct packet manipulation.
This approach successfully parses destination addresses, overwrites them, and recalculates the required header checksums using inline bitwise arithmetic directly within the driver hook.
When combined with direct egress redirection, this creates a rudimentary but exceptionally high-speed software switch.
While XDP-L2-Guard is not a complete, production-ready router, it successfully demonstrates a highly impactful fraction of eBPF's capabilities, confirming that Linux possesses the architectural potential to handle complex, high-speed routing and rewriting at the lowest possible software level.

\section{Project Summary and Conclusions}

\subsection{Impressions and Lessons Learned}

Executing the XDP-L2-Guard project provided invaluable hands-on experience with the lowest levels of the Linux networking stack.
Our primary impression is one of profound capability mixed with steep learning curves.
eBPF and XDP offer unprecedented power to intercept and modify packets at wire speed, but this power comes at the cost of strict Verifier constraints and complex C-level memory handling.
We successfully proved the viability of designing an autonomous data plane capable of parsing network headers, rewriting checksums, and orchestrating high-speed actions before the kernel allocates standard socket buffers.

\subsection{Why This Project Was Worth Doing}

This project was highly valuable because it demystified the mechanisms behind modern cloud-native networking and software-defined networking (SDN).
By building XDP-L2-Guard from scratch, we shifted our understanding of Linux from a simple endpoint or basic firewall into a highly programmable, hardware-adjacent switch.
It allowed us to practically experience the limitations of legacy tools like \texttt{iptables} and fully grasp why the industry is rapidly adopting eBPF.
The knowledge gained regarding Native XDP, memory bounds-checking, and control/data plane separation forms a critical foundation for modern network engineering.

\subsection{Future Research and Testing Directions}

While XDP-L2-Guard serves as a successful proof-of-concept, there are numerous avenues for potential future exploration:
\begin{itemize}
    \item \textbf{Deep Layer-2/Label Parsing:} Expanding the data plane to natively parse, push, and pop MPLS labels or VLAN tags to simulate a complete Carrier Ethernet Label Switch Router (LSR).
    \item \textbf{Advanced Hardware Tuning:} Further resolving the low-level hardware configuration nuances of Intel NICs to fully unleash Native Mode (\texttt{xdpdrv}) and benchmark it against DPDK-based solutions at 40+ Gbps line rates.
    \item \textbf{Stateful Tracking:} Transitioning from stateless DNAT to fully stateful connections by implementing connection tracking (conntrack) utilizing eBPF LRU (Least Recently Used) hash maps.
    \item \textbf{IPv6 Support:} Extending the C codebase and bounds-checking logic to comprehensively handle IPv6 headers and ICMPv6 routing protocols.
\end{itemize}

\subsection{Data Plane Deep Dive: eBPF/XDP C Implementation}

The kernel data plane resides in \texttt{src/data\_plane/filter.c} and \texttt{src/data\_plane/headers.h}.

\subsubsection{Structural Definitions \& Memory Bounds Checking}

Because eBPF programs run in kernel space, any invalid memory access would be catastrophic.
The eBPF Verifier requires mathematical proof that all packet accesses fall within valid memory ranges.
To facilitate this, \texttt{headers.h} defines core networking structures and the critical \texttt{BOUNDS\_CHECK} macro.

\begin{lstlisting}[language=C, title={src/data\_plane/headers.h - Core Macros \& Structures}]
#ifndef __HEADERS_H
#define __HEADERS_H

#include <linux/bpf.h>
#include <linux/if_ether.h>
#include <linux/ip.h>
#include <linux/in.h>
#include <bpf/bpf_helpers.h>
#include <bpf/bpf_endian.h>

struct icmphdr {
  __u8		type;
  __u8		code;
  __sum16	checksum;
  union {
	struct {
		__be16	id;
		__be16	sequence;
	} echo;
	__be32	gateway;
	struct {
		__be16	__unused;
		__be16	mtu;
	} frag;
	__u8	reserved[4];
  } un;
};

#define ICMP_ECHO		8
#define ICMP_ECHOREPLY		0

#define BOUNDS_CHECK(pointer, type, data_end) \
    do { \
        if ((void *)(pointer) + sizeof(type) > (void *)(data_end)) \
            return XDP_PASS; \
    } while(0)

#endif // __HEADERS_H
\end{lstlisting}

\noindent \textbf{Code Analysis:} The \texttt{BOUNDS\_CHECK} macro takes a given memory pointer, evaluates its structural size using \texttt{sizeof(type)}, and compares the resulting boundary against \texttt{data\_end}.
If the calculated pointer extends beyond the packet buffer, the macro instantly forces an \texttt{XDP\_PASS}, allowing the Linux kernel to handle or drop the malformed frame.
This guarantees that the eBPF Verifier passes the program without throwing pointer arithmetic errors.

#include "headers.h"

enum xdp_action_target {
    ACTION_PASS = 0,
    ACTION_DROP,
    ACTION_TX,
    ACTION_REDIRECT,
    ACTION_NAT
};

struct action_cfg {
    __u32 target;           // Selected action (PASS, DROP, TX, REDIRECT, NAT)
    __u32 new_ip;           // Replacement Destination IP for NAT actions
    __u32 ifindex;          // Output network interface index for REDIRECT
    __u64 dropped_packets;  // Atomic per-rule drop counter
};

struct {
    __uint(type, BPF_MAP_TYPE_HASH);
    __type(key, __u32);
    __type(value, struct action_cfg);
    __uint(max_entries, 10000);
} action_map SEC(".maps");

struct {
    __uint(type, BPF_MAP_TYPE_DEVMAP);
    __uint(key_size, sizeof(__u32));
    __uint(value_size, sizeof(__u32));
    __uint(max_entries, 256);
} dev_map SEC(".maps");

struct {
    __uint(type, BPF_MAP_TYPE_ARRAY);
    __type(key, __u32);
    __type(value, __u64);
    __uint(max_entries, 1);
} total_dropped SEC(".maps");

struct {
    __uint(type, BPF_MAP_TYPE_HASH);
    __type(key, __u32);
    __type(value, __u64);
    __uint(max_entries, 10000);
} source_stats_map SEC(".maps");
\end{lstlisting}

\noindent \textbf{Code Analysis:} The \texttt{action\_map} is a high-performance hash table storing up to 10,000 filtering directives.
The key is an IPv4 address stored as a 32-bit unsigned integer (\texttt{\_\_u32}) in network byte order.
The value is a \texttt{struct action\_cfg} containing the execution target, a replacement IP for NAT operations, an egress interface index (\texttt{ifindex}) for redirection, and an atomic packet drop counter.
\texttt{dev\_map} is a specialized BPF map (\texttt{BPF\_MAP\_TYPE\_DEVMAP}) used exclusively by the \texttt{bpf\_redirect\_map} helper to forward frames to hardware interface rings.
\texttt{source\_stats\_map} tracks real-time packet ingress volumes per source IP for telemetry dashboards.

static __always_inline void swap_mac(struct ethhdr *eth) {
    __u8 tmp[ETH_ALEN];
    __builtin_memcpy(tmp, eth->h_source, ETH_ALEN);
    __builtin_memcpy(eth->h_source, eth->h_dest, ETH_ALEN);
    __builtin_memcpy(eth->h_dest, tmp, ETH_ALEN);
}

static __always_inline void swap_ip(struct iphdr *ip) {
    __u32 tmp = ip->saddr;
    ip->saddr = ip->daddr;
    ip->daddr = tmp;
}

static __always_inline void update_ip_checksum(struct iphdr *ip, __u32 new_daddr) {
    __u32 old_daddr = ip->daddr;
    __u32 csum = ~bpf_ntohs(ip->check) & 0xFFFF;
    csum += (~(old_daddr & 0xFFFF) & 0xFFFF) + (new_daddr & 0xFFFF);
    csum += (~(old_daddr >> 16) & 0xFFFF) + (new_daddr >> 16);
    while (csum >> 16) csum = (csum & 0xFFFF) + (csum >> 16);
    ip->check = bpf_htons(~csum & 0xFFFF);
    ip->daddr = new_daddr;
}
\end{lstlisting}

\noindent \textbf{Code Analysis:} The \texttt{\_\_always\_inline} directive forces the compiler to embed these functions directly into the main execution path, preventing function call overhead in kernel space.
\texttt{swap\_mac} utilizes the compiler built-in \texttt{\_\_builtin\_memcpy} to swap the 6-byte source and destination MAC addresses efficiently.
\texttt{update\_ip\_checksum} avoids a computationally expensive loop across the entire IP header.
Instead, it leverages the mathematical properties of 1's complement arithmetic.
It subtracts the old destination IP (by adding its bitwise negation) and adds the new destination IP in 16-bit words.
Any carry bits overflowing the 16-bit boundary are folded back into the lower 16 bits until no overflow remains.
Finally, the value is inverted and converted back to big-endian network byte order via \texttt{bpf\_htons}.

SEC("xdp")
int xdp_drop_logic(struct xdp_md *ctx) {
    void *data = (void *)(long)ctx->data;
    void *data_end = (void *)(long)ctx->data_end;

    struct ethhdr *eth = data;
    BOUNDS_CHECK(eth, struct ethhdr, data_end);

    if (eth->h_proto != bpf_htons(ETH_P_IP)) return XDP_PASS;
    struct iphdr *ip = data + sizeof(struct ethhdr);
    BOUNDS_CHECK(ip, struct iphdr, data_end);

    __u32 src_ip = ip->saddr;
    __u64 *pkt_count = bpf_map_lookup_elem(&source_stats_map, &src_ip);
    if (pkt_count) {
        __sync_fetch_and_add(pkt_count, 1);
    } else {
        __u64 init_count = 1;
        bpf_map_update_elem(&source_stats_map, &src_ip, &init_count, BPF_ANY);
    }

    struct action_cfg *cfg = bpf_map_lookup_elem(&action_map, &src_ip);
    
    bpf_printk("XDP: lookup src_ip=%pI4 hex=%x cfg=%p", &src_ip, src_ip, cfg);
    if (!cfg) return XDP_PASS;

    switch (cfg->target) {
        case ACTION_DROP: {
            __u32 key = 0;
            __u64 *val = bpf_map_lookup_elem(&total_dropped, &key);
            if (val) __sync_fetch_and_add(val, 1);
            __sync_fetch_and_add(&cfg->dropped_packets, 1);
            return XDP_DROP;
        }

        case ACTION_TX: {
            if (ip->protocol == IPPROTO_ICMP) {
                struct icmphdr *icmp = (void *)ip + sizeof(struct iphdr);
                BOUNDS_CHECK(icmp, struct icmphdr, data_end);

                if (icmp->type == ICMP_ECHO) {
                    icmp->type = ICMP_ECHOREPLY;
                    __u32 temp_csum = bpf_ntohs(icmp->checksum) + 0x0800;
                    if (temp_csum > 0xFFFF) temp_csum -= 0xFFFF;
                    icmp->checksum = bpf_htons(temp_csum);

                    swap_mac(eth);
                    swap_ip(ip);
                    return XDP_TX;
                }
            }
            return XDP_PASS;
        }

        case ACTION_REDIRECT: {
            if (cfg->new_ip) {
                update_ip_checksum(ip, cfg->new_ip);
            }
            int err = bpf_redirect_map(&dev_map, cfg->ifindex, 0);
            bpf_printk("XDP: redirect dnat=%pI4 ifindex=%u res=%d", &cfg->new_ip, cfg->ifindex, err);
            return err;
        }

        case ACTION_NAT: {
            update_ip_checksum(ip, cfg->new_ip);
            return XDP_PASS;
        }

        case ACTION_PASS:
        default:
            return XDP_PASS;
    }
}

\noindent \textbf{Code Analysis:} 
\begin{itemize}
    \item \textbf{Ingress Telemetry:} Before evaluating filtering rules, the code extracts \texttt{ip->saddr}.
It queries \texttt{source\_stats\_map} via \texttt{bpf\_map\_lookup\_elem}. If an entry exists, it performs an atomic increment using \texttt{\_\_sync\_fetch\_and\_add(pkt\_count, 1)}.
If not, it registers the IP with an initial count of 1. This happens at wire speed for every incoming IPv4 packet.
    \item \textbf{Rule Evaluation:} The source IP is looked up in \texttt{action\_map}.
If no configuration (\texttt{cfg}) is found, the packet defaults to \texttt{XDP\_PASS}, handing it over to the standard Linux network stack.
    \item \textbf{\texttt{ACTION\_DROP}:} If the target is \texttt{ACTION\_DROP}, the global drop counter (\texttt{total\_dropped}) and the per-rule drop counter (\texttt{cfg->dropped\_packets}) are atomically incremented.
The function returns \texttt{XDP\_DROP}, instructing the NIC driver to drop the frame and recycle the DMA buffer instantly.
    \item \textbf{\texttt{ACTION\_TX} (ICMP Hairpinning):} If the packet is an ICMP echo request (Ping, Type 8), it is converted to an echo reply (Type 0).
Because the type field changes by a value of 8 (\texttt{0x0800} in 16-bit big-endian representation), the checksum is adjusted by adding \texttt{0x0800}.
The MAC and IP addresses are swapped, and \texttt{XDP\_TX} bounces the packet back out of the incoming interface, completely bypassing the OS routing table.
    \item \textbf{\texttt{ACTION\_REDIRECT}:} If a new destination IP is specified, \texttt{update\_ip\_checksum} rewrites the packet.
Then, \texttt{bpf\_redirect\_map} forwards the frame to the interface index defined in \texttt{dev\_map}.
\end{itemize}

For \texttt{REDIRECT} actions, it dynamically resolves the interface name (e.g., \texttt{veth1}) to its kernel interface index (\texttt{ifindex}) using \texttt{ip -o link show dev}.
To populate the \texttt{action\_map}, the Python script must format data to match the C \texttt{struct action\_cfg} layout exactly.
Because C compilers insert padding to align 64-bit integers (\texttt{\_\_u64 dropped\_packets}) on 8-byte boundaries, Python's \texttt{struct.pack("<IIIIQ", ...)} explicitly packs four 32-bit integers (target, new\_ip, ifindex, padding) followed by a 64-bit integer (dropped\_packets).
This produces a perfectly aligned 24-byte hex string that \texttt{bpftool map update} injects into kernel memory.

\noindent \textbf{Code Analysis:} Due to a strict 15-character limit on eBPF map names in older kernel versions, \texttt{source\_stats\_map} is truncated in kernel space to \texttt{source\_stats\_ma}. The script retrieves this map ID using \texttt{bpftool}. In a 1-second polling loop, it extracts the map contents in JSON format, decodes the hex keys into IPv4 strings using \texttt{socket.inet\_ntoa}, unpacks the 64-bit unsigned integer values (\texttt{<Q}) representing packet counts, sorts the results in descending order, and 
clears the terminal screen to display a dynamic, real-time telemetry dashboard.

\subsubsection{Testbench Topology Setup}

The \texttt{setup.sh} script establishes a simulated 3-node network environment using virtual Ethernet pairs:
\begin{itemize}

    \item \textbf{\texttt{ns2} (Sender)}: Assigned IP \texttt{10.0.0.2}. Acts as the source of traffic.
    \item \textbf{\texttt{ns3} (Target)}: Assigned IP \texttt{10.0.0.3} in promiscuous mode to observe redirected packets.
\end{itemize}

The rule is injected into kernel memory instantly, without interrupting network connections or requiring a system reboot.

\subsubsection{Phase 4: Real-Time Kernel-Space Packet Dropping}

\begin{figure}[H]
    \centering
    \includegraphics[width=0.9\textwidth]{images/screenshot_ping_dropped.png}
    \caption{Enforcing the XDP policy: Immediate 100\% packet loss as XDP\_DROP intercepts frames at the driver level.}
\end{figure}

\noindent \textbf{Empirical Walkthrough:} The moment the rule is injected into \texttt{action\_map}, the execution path inside \texttt{xdp\_drop\_logic} changes.
As incoming ICMP packets arrive from \texttt{10.0.0.2}, \texttt{bpf\_map\_lookup\_elem} locates the matching configuration with \texttt{target == ACTION\_DROP}.
The program executes \texttt{return XDP\_DROP}. The network driver drops the frames directly from the RX ring buffer before the Linux kernel is aware of them.
The screenshot demonstrates this real-time impact: the ping output abruptly stops receiving replies, showing 100\% packet loss.
Simultaneously, the polling loop in \texttt{loader.py} outputs active drop notifications, confirming that the atomic counters are incrementing at wire speed.

\subsubsection{Phase 5: Fast-Path Egress Redirection \& Stateless NAT}

\begin{figure}[H]
    \centering
    \includegraphics[width=0.9\textwidth]{images/screenshot_redirect.png}
    \caption{XDP\_REDIRECT in action: Traffic intended for 10.0.0.1 is transparently intercepted, rewritten, and routed to the target namespace (ns3) at the driver level.}
\end{figure}

\noindent \textbf{Empirical Walkthrough:} To evaluate advanced data plane forwarding, the operator replaces the drop rule with a redirection policy: \texttt{guard.sh -A -d 10.0.0.2 -j REDIRECT --oif v1-3}.
When packets arrive at \texttt{v1-2}, \texttt{xdp\_drop\_logic} identifies the \texttt{ACTION\_REDIRECT} target.
The helper function \texttt{update\_ip\_checksum} rewrites the destination IP and updates the header checksum using 1's complement bitwise arithmetic.
The program then calls \texttt{bpf\_redirect\_map(\&dev\_map, cfg->ifindex, 0)}, forwarding the raw frame directly out of interface \texttt{v1-3} toward \texttt{ns3}.
The screenshot confirms this end-to-end operation. In the top terminal, \texttt{ns2} transmits packets toward \texttt{10.0.0.1}.
In the bottom terminal, a packet sniffer running in promiscuous mode inside \texttt{ns3} (\texttt{10.0.0.3}) captures the redirected frames.
The entire interception, rewriting, and forwarding lifecycle occurs entirely within the eBPF layer at the NIC driver hook, completely bypassing the standard routing tables and firewall rules in \texttt{ns1}.
